# MS 업무자동화 솔루션 컨설팅 CLI 에이전트 요구사항 문서

## 1. 문서 개요

### 1.1 프로젝트명
MS 업무자동화 솔루션 컨설팅 CLI 에이전트

### 1.2 문서 목적
본 문서는 사용자가 VSCode 환경에서 Codex CLI를 통해 업무 자동화 관련 요구사항을 입력하면, AI가 이를 분석하여 Microsoft Software 중심의 자동화 방안을 제안하고, 여러 AI의 의견 비교 및 최종 산출물 파일 생성을 지원하는 프로젝트의 요구사항을 정의한다.

| v1.0 | 2026-03-06 | 최초 작성 |

## 2. 프로젝트 목표

### 2.1 핵심 목표

- 사용자의 업무 자동화 요구사항을 구조적으로 분석한다.
- Microsoft Software 중심으로 적절한 자동화 방안을 제안한다.
- GPT, Gemini, Claude의 의견을 비교하고, **근거 중심의 단일 최종 권고안 + 대안** 형식으로 제공한다.
- 사용자가 최종안을 선택하면 **비개발자용 요약 섹션 + 개발자용 상세 섹션**으로 구성된 산출물 파일을 생성한다.
- MS 솔루션 지식 베이스를 **MS Graph API + Power Platform 공식 문서를 RAG로 연동**하여 최신성을 확보한다.
- 이 프로젝트 개발 과정에서 발생하는 에러/수정/변경 이유를 별도 로깅 구조로 관리한다.

## 4. 사용자 시나리오

### 4.1 기본 시나리오

1. 사용자가 VSCode에서 Codex CLI를 실행한다.
2. 사용자가 업무 자동화 요구사항을 프롬프트 형태로 입력한다.
3. 시스템은 요구사항을 분석하고 업무 단계를 구조화한다.
4. 시스템은 MS 라이선스/정책 제약 조건을 사전 필터링한 뒤 Microsoft Software 중심 자동화 방안을 제안한다.
5. 시스템은 선택된 모드(Quick / Deep)에 따라 AI 의견을 수집하고 비교 결과를 터미널에 표시한다.
6. 시스템은 가중치 점수표 기반으로 **근거 중심 단일 권고안 + 대안**을 제시한다.
7. 사용자가 최종 선택을 입력한다.
8. 시스템은 선택 결과를 기반으로 **비개발자용 요약 + 개발자용 상세** 두 섹션으로 구성된 산출물 파일을 생성한다.

### 4.2 보완 시나리오

- 요구사항이 불명확한 경우 누락된 항목 또는 확인이 필요한 질문을 제시한다.
- 하나의 솔루션이 아닌 복수 대안을 비교 형태로 보여준다.
- 사용자가 txt 또는 docx 등 원하는 파일 형식을 선택할 수 있다.
- AI 호출 일부 실패 시 Fallback 모드로 자동 전환되며, 사용자에게 품질 저하 여부를 명시한다.

### 5.2 MS 솔루션 추천 및 지식 소스 분리

#### 5.2.1 솔루션 추천
- Microsoft 제품군(Power Automate, Teams, SharePoint, Power BI, Azure 등) 중심으로 솔루션을 제안한다.
- MS 라이선스 및 정책 제약을 Requirement Analyzer 단계에서 사전 필터링한다.

#### 5.2.2 지식 소스 이원화 아키텍처

두 가지 지식 소스를 역할에 따라 분리하여 운영한다.

| 채널 | 대상 | 구현 방법 |
|---|---|---|
| **MS Graph API** | 조직 컨텍스트 (사용자/팀/라이선스/권한 정보) | OAuth 2.0 인증 기반 API 호출 |
| **Power Platform RAG** | 제품·커넥터·정책 최신 지식 | MS 공식 문서 인덱싱 + 벡터 검색 |

**KnowledgeRouter**: 질의 의도를 분류하여 MS Graph 채널 또는 Power Platform RAG 채널로 라우팅한다.

**ContextMerger**: 두 채널의 검색 결과를 병합하고 출처를 명시하여 LLM에 전달한다.

> 리스크: MS Graph OAuth 토큰 관리 필요, RAG 인덱스의 최신성 유지를 위한 정기 크롤링 일정 수립 필요

### 5.3 AI 의견 비교 및 권고안 도출

#### 5.3.1 AI 의견 수집 모드 분리

사용자는 실행 시 `--mode` 플래그 또는 대화형 메뉴로 모드를 선택한다.

| 모드 | 목표 응답 시간 | 구성 | 용도 |
|---|---|---|---|
| **Quick** | 20~40초 | 단일 LLM + 캐시된 MS 지식 | 빠른 초안 확인 |
| **Deep** | 60~120초 | GPT/Gemini/Claude 3종 병렬 + RAG 풀 파이프라인 | 중요 의사결정, 상세 비교 |
| **Fallback** | 상황에 따라 가변 | 성공한 1~2개 모델로 자동 전환 | 일부 AI 호출 실패 시 |

- asyncio 기반 비동기 오케스트레이터로 병렬 수집을 수행한다.
- Fallback 모드 전환 시 사용자에게 "일부 모델 응답 불가 - 품질이 제한될 수 있습니다" 메시지를 표시한다.

#### 5.3.2 가중치 점수표 기반 단일 권고안 도출

각 AI 제안 솔루션을 아래 기준으로 정량 평가하여 최고 점수 안을 권고안으로 채택한다.

| 평가 항목 | 가중치 | 채점 기준 (1~5점) |
|---|---|---|
| 적합성 | 35% | 업무 요구사항 부합 정도 |
| 구현 난이도 | 30% | 낮을수록 고점 (1=매우 어려움, 5=매우 쉬움) |
| 유지보수성 | 15% | 운영·변경 용이성 |
| 라이선스 적합성 | 10% | 조직 보유 라이선스 범위 내 구현 가능 여부 |
| 확장성 | 10% | 향후 기능 추가 용이성 |
| **합계** | **100%** | |

> 주의: 동점 발생 시 적합성 > 구현 난이도 순으로 우선 처리

**구현 방법**:
- 각 AI는 평가 항목별로 점수(1~5)와 채택 근거를 JSON 형식으로 출력한다.
- Recommendation Engine이 가중합을 계산하고 1위(권고안) + 2위(대안) 순위를 결정한다.
- 채점 루브릭(명확한 기준 설명)을 프롬프트에 포함하여 모델 간 점수 편차를 최소화한다.
- Pydantic 모델 `ScoringResult`로 타입 안전성을 보장한다.

```python
# 예시 데이터 구조
class ScoringResult(BaseModel):
    solution_name: str
    scores: dict[str, float]   # 항목명: 점수(1~5)
    rationale: dict[str, str]  # 항목명: 근거 텍스트
    weighted_total: float      # 가중합 최종 점수
```

- 평가 결과와 가중합 계산 과정을 Dev Log Manager에 기록하여 감사 추적이 가능하도록 한다.

### 5.4 산출물 파일 생성 — 이중 섹션 구조

사용자가 선택한 최종안을 기반으로 **단일 파일 내 두 섹션**으로 산출물을 생성한다.

#### 섹션 구성

| 섹션 | 독자 | 내용 |
|---|---|---|
| **[Part 1] 비개발자용 요약** | 관리자, 업무 담당자 | 자동화 목적, 제안 솔루션 요약, 기대 효과, 도입 단계 로드맵 |
| **[Part 2] 개발자용 상세 명세** | 개발자, 시스템 담당자 | 모듈 구조, API 연동 방법, 데이터 흐름, 가중치 채점 결과, 구현 시 주의사항 |

#### Part 1 — 비개발자용 요약 구성 항목
1. 프로젝트 개요 (한 줄 요약)
2. 현재 업무 문제점
3. 제안 자동화 솔루션 (MS 제품군 중심)
4. 기대 효과 (시간 절감, 오류 감소 등 정량 표현 권장)
5. 도입 단계 (단계별 로드맵, 예: 1단계 파이럿 / 2단계 확산)
6. 주의사항 및 전제 조건 (라이선스, 조직 승인 필요 항목 등)

#### Part 2 — 개발자용 상세 명세 구성 항목
1. 채택 솔루션 기술 스펙 (API, 커넥터, 연동 방법)
2. 시스템 아키텍처 다이어그램 (텍스트 기반 ASCII 또는 Mermaid)
3. 모듈별 역할 및 인터페이스 (Pydantic 모델 기준)
4. 가중치 점수표 전체 결과 (항목별 점수 + 근거)
5. AI 의견 비교 테이블 (Quick/Deep 모드 결과)
6. 구현 시 주의사항 (인증, 토큰 관리, 에러 처리 등)
7. 테스트 항목 리스트 (5.6 유저 테스트 skill 연동)

#### 지원 포맷 및 설계 원칙
- 지원 포맷: `txt`, `docx`
- File Generator는 **어댑터 패턴**으로 설계하여 추후 Notion 등 다른 포맷 연동 시 코드 변경 최소화
- 두 섹션 동기화 이슈를 방지하기 위해 동일 데이터 소스에서 자동 생성 (수동 편집 지양)

### 5.6 유저 테스트 리스트 자동 생성 Skill

개발 완료 후 테스트 단계에서 UserRequirement.md를 파싱하여 검증에 필요한 테스트 항목을 자동으로 생성하는 skill이다.

#### 동작 방식
1. UserRequirement.md의 기능 요구사항(5.x 항목)을 파싱한다.
2. 각 기능 단위에 대해 LLM에 **Given-When-Then** 형식의 프롬프트를 제공하여 테스트 시나리오를 생성한다.
3. 생성된 테스트 케이스를 `기능명 / 테스트 조건 / 기대 결과 / 우선순위` 구조의 Markdown 체크리스트로 출력한다.

#### 출력 예시
```markdown
## 유저 테스트 체크리스트

### [5.3] AI 의견 비교 및 권고안 도출
- [ ] **[P1] Deep 모드 정상 작동**: 3개 AI 모두 응답할 때 가중합 기준 1위 권고안이 출력되는가?
- [ ] **[P1] Fallback 모드 전환**: Gemini 호출 실패 시 2개 모델만으로 결과가 생성되고 경고 메시지가 표시되는가?
- [ ] **[P2] 가중치 합산 검증**: 점수표 합산이 항상 100%인가?

### [5.4] 산출물 파일 생성
- [ ] **[P1] docx 생성 검증**: 비개발자 섹션과 개발자 섹션이 모두 포함된 파일이 생성되는가?
- [ ] **[P2] 포맷 선택**: txt / docx 각각 정상 생성되는가?
```

#### 제약 및 주의사항
- 요구사항 문서 품질이 낮으면 테스트 케이스 누락 또는 중복 발생 가능
- 자동 생성 결과에 대해 **human review 게이트 필수** (자동 생성은 초안으로만 사용)
- 개발 완료 시점에서 실행하며, 코드베이스의 실제 모듈명/함수명을 반영하여 재생성 가능

### 5.7 토큰 소모량 최적화 전략

3개 AI 병렬 호출 및 RAG 연동으로 인한 토큰 소모를 줄이기 위해 아래 전략을 적용한다.

| 전략 | 내용 |
|---|---|
| **모드 분리** | 기본값은 Quick 모드(단일 모델)로 실행, Deep 모드는 명시적 선택 시만 활성화 |
| **프롬프트 캐싱** | 동일 시스템 프롬프트 재사용으로 반복 입력 토큰 절감 |
| **RAG 청크 요약 전달** | 검색된 청크 원문 대신 핵심 요약본만 LLM 컨텍스트로 전달 |
| **모델별 역할 분리** | Deep 모드에서 각 모델에 역할 부여 (요약 / 비평 / 결론) — 전체 분석을 3배 반복하지 않음 |
| **중간 결과 캐싱** | 동일 요구사항 재질의 시 이전 AI 응답 재사용 |
| **max_tokens 제한** | 모델별 응답 토큰 상한 설정 (과도한 장문 응답 방지) |

> 주의: 압축·요약 과정에서 근거 손실이 발생하면 권고안 품질 저하 가능 — 요약 수준과 품질 간 균형 모니터링 필요

| 응답 시간 (Deep) | 60~120초 이내 |
| 응답 시간 (Fallback) | 성공 모델 수에 따라 가변, 사용자에게 상태 표시 |
| 확장성 | 출력 어댑터 패턴으로 신규 포맷 추가 가능 |
| 신뢰성 | AI 호출 실패 시 Fallback 모드로 자동 전환, 사용자에게 품질 저하 명시 |
| 유지보수성 | 모듈 간 표준 인터페이스(Pydantic 모델)로 독립성 보장 |
| 보안 | MS Graph OAuth 토큰 안전 관리, 개발자용 섹션 외부 공유 시 별도 파일로 분리 고려 |

## 7. 시스템 구조 요구사항

### 7.1 권장 구조

초기 단계에서는 실제 에이전트 런타임보다 **모듈형 오케스트레이션 구조**로 시작한다.

### 7.2 모듈 구성

| 모듈 | 역할 |
|---|---|
| **Input Parser** | 사용자 입력 파싱 및 구조화 |
| **Requirement Analyzer** | 요구사항 분석, MS 라이선스/정책 제약 사전 필터링 |
| **KnowledgeRouter** | 질의 의도 분류 → MS Graph 또는 Power Platform RAG 채널 라우팅 |
| **MS Graph Connector** | 조직 컨텍스트(사용자/팀/라이선스) 조회 (OAuth 2.0) |
| **Power Platform RAG** | Power Platform·MS 공식 문서 벡터 검색 파이프라인 |
| **ContextMerger** | 두 지식 소스 결과 병합 및 출처 명시 |
| **AI Comparison Orchestrator** | Quick/Deep/Fallback 모드로 GPT/Gemini/Claude 의견 병렬 수집 (asyncio) |
| **Recommendation Engine** | 가중치 점수표 기반 권고안 + 대안 도출, ScoringResult 생성 |
| **Output Formatter** | 결과 포맷팅 (터미널 출력) |
| **File Generator** | 비개발자용+개발자용 이중 섹션 산출물 생성 (어댑터 패턴) |
| **Test Skill** | UserRequirement.md 파싱 → 유저 테스트 체크리스트 자동 생성 |
| **Dev Log Manager** | structlog + JSONL 기반 에러/변경/채점 이력 로깅 |

### 7.3 모듈 간 인터페이스
- 모듈 간 데이터는 **Pydantic 모델 (또는 JSON 스키마)**로 표준화하여 독립성 및 테스트 가능성을 확보한다.
- 핵심 데이터 모델: `RequirementModel`, `SolutionProposal`, `ScoringResult`, `OutputDocument`

### 7.4 아키텍처 흐름 (텍스트 다이어그램)

```
[User Input]
     │
     ▼
[Input Parser] ──► [Requirement Analyzer]
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
   [MS Graph Connector]    [Power Platform RAG]
              └───────────┬───────────┘
                          ▼
                   [ContextMerger]
                          │
                          ▼
            [AI Comparison Orchestrator]
             Quick / Deep / Fallback 모드
                          │
                          ▼
              [Recommendation Engine]
              가중치 점수표 → 권고안 + 대안
                          │
                          ▼
               [Output Formatter] ──► 터미널 출력
                          │
                          ▼
                  [File Generator]
          Part1(비개발자) + Part2(개발자) 섹션
                          │
                          ▼
                    산출물 파일 (.txt / .docx)

[Dev Log Manager] ◄── 전 모듈에서 로그 수집
[Test Skill] ──► UserRequirement.md 파싱 → 테스트 체크리스트
```

## 9. 용어 정의

| 용어 | 정의 |
|---|---|
| RAG | Retrieval-Augmented Generation. 외부 문서를 검색하여 LLM 응답의 정확성을 높이는 기법 |
| MS Graph API | Microsoft 365 서비스(Teams, SharePoint, OneDrive 등)에 접근하는 통합 API |
| Power Platform | Power Automate, Power Apps, Power BI, Power Virtual Agents를 포함하는 MS 저코드 플랫폼 |
| KnowledgeRouter | 질의 의도를 분류하여 MS Graph 또는 Power Platform RAG 채널로 라우팅하는 컴포넌트 |
| ContextMerger | 두 지식 소스(MS Graph + Power Platform RAG)의 결과를 병합하는 컴포넌트 |
| ScoringResult | 가중치 점수표 평가 결과를 담는 Pydantic 데이터 모델 |
| 어댑터 패턴 | 출력 포맷(txt, docx, Notion 등)을 인터페이스로 추상화하여 확장 가능하게 만드는 설계 패턴 |
| Quick 모드 | 단일 LLM + 캐시 지식으로 20~40초 내 결과를 제공하는 빠른 실행 모드 |
| Deep 모드 | 3개 AI 병렬 + RAG 풀 파이프라인으로 60~120초 내 상세 결과를 제공하는 모드 |
| Fallback 모드 | 일부 AI 호출 실패 시 성공한 모델만으로 자동 진행하는 복구 모드 |
| Dev Log Manager | 개발 중 발생하는 에러, 수정 이유, 변경 내역 및 채점 이력을 기록하는 모듈 |
| Given-When-Then | BDD(행동 주도 개발)에서 사용하는 테스트 시나리오 작성 형식 |
