# 재컨설팅 타입 판정 가이드

`consult` 스킬에서 사용자 피드백을 A/B/C 타입으로 분류하는 기준과 예시.

---

## 판정 핵심 원칙

```
솔루션 ID(핵심 MS 제품 조합)가 바뀌는가?
  YES → 타입 B
  NO  → 타입 A
설명/질문만인가?
  YES → 타입 C
```

**솔루션 ID 예시**: `PowerAutomate+SharePointList+Outlook`
- 핵심 제품 조합이 달라지면 ID 변경
- 운영 방식/파라미터만 바뀌면 ID 유지

---

## 타입 A — 경량 재컨설팅

**정의**: 현재 솔루션 ID 유지. 조건·파라미터·세부 구성만 조정.

### 명확한 타입 A 예시

| 피드백 | 이유 |
|---|---|
| "발송 주기를 매일로 바꿔줘" | 제품 조합 유지, 스케줄만 변경 |
| "SharePoint 대신 OneDrive로 바꿔줘" | 보조 저장소 변경, 핵심 제품(Power Automate) 유지 |
| "첨부파일 용량 제한 조건 추가해줘" | 파라미터 추가 |
| "발송 실패 시 재시도 로직 추가해줘" | 기존 흐름 내 옵션 추가 |
| "Teams 알림도 같이 붙여줘" | 현재 솔루션에 알림 채널 추가 |
| "이메일 수신자를 내부로만 제한하고 싶어" | 조건 변경, 제품 변경 없음 |

---

## 타입 B — 전체 재컨설팅

**정의**: 솔루션 ID가 변경될 수밖에 없는 변경. 전체 재분석 권장.

### 명확한 타입 B 예시

| 피드백 | 이유 |
|---|---|
| "Power Automate 말고 다른 걸로" | 핵심 제품 교체 → ID 변경 |
| "DLP 정책으로 Power Automate 못 써" | 핵심 제품 차단 → 전체 재분석 필요 |
| "Azure Logic Apps 방향으로 바꾸고 싶어" | 핵심 제품 완전 교체 |
| "라이선스가 없어서 이 솔루션 못 써" | 솔루션 자체 불가 → 대안 필요 |
| "요구사항을 완전히 바꾸고 싶어" | 전제 조건 변경 |
| "비용을 최소화해야 해서 다른 방향으로" | 솔루션 선택 기준 변경 |

---

## 타입 C — 설명 요청

**정의**: 현재 권고안 변경 없이 설명·질문만 요청. 재컨설팅 없음.

### 명확한 타입 C 예시

| 피드백 | 이유 |
|---|---|
| "이 리스크가 무슨 뜻이야?" | 설명 요청 |
| "구현 개요 더 자세히 설명해줘" | 상세 설명 요청 |
| "DLP 정책이 뭐야?" | 용어 설명 |
| "이 방식의 장점이 뭐야?" | 비교/이유 질문 |
| "구현하면 얼마나 걸려?" | 예측 질문 (설명으로 응답) |
| "왜 SharePoint를 썼어?" | 선택 이유 질문 |

## 경계 케이스 — 애매한 경우 판정 기준

### "조금만 바꿔줘" 류

```
→ 어떤 부분을 바꾸는지 먼저 파악
→ 핵심 제품이 그대로면 A / 바뀌면 B
모르면 질문: "어떤 부분을 바꾸고 싶으신가요?"
```

### "더 간단한 방법 없어?" 류

```
→ 현재 솔루션의 단순화 = 타입 A
→ 완전히 다른 접근 방식 요구 = 타입 B
판단 불명확하면 B로 분류
```

### "비용이 걱정돼" 류

```
→ 현재 솔루션 내 비용 최적화 = 타입 A
→ 다른 솔루션으로 변경 요청 = 타입 B
명확하지 않으면: "현재 솔루션을 유지하면서 비용을 줄이는 방법을 찾을까요,
아니면 다른 솔루션으로 바꿀까요?" 질문 후 판정
```

### "라이선스 문제가 있어" 류

```
→ 라이선스 취득 가능 여부 확인 후 유지 = 타입 A (또는 C)
→ 라이선스 취득 불가 → 솔루션 변경 필요 = 타입 B
판단 불명확하면 먼저 타입 C로 설명, 이후 상황에 따라 B 전환
```

### "Teams 알림"처럼 새 제품 추가 요청

```
→ 기존 핵심 제품 유지 + 보조 채널 추가 = 타입 A
→ 기존 핵심 제품이 Teams로 교체 = 타입 B
기준: 기존 솔루션 ID의 핵심 제품이 그대로 남아 있는가?
```

### 설명 + 변경 혼합 요청

```
예) "이 리스크 설명해주고, 가능하면 대응 방안도 추가해줘"
→ 설명 = 타입 C로 먼저 응답
→ 대응 방안 추가 = 타입 A로 이어서 처리
복합 요청은 순서대로 분리해서 처리
```

## 타입 B — user_override 판정

사용자가 타입 B 확인 요청에 "아니오(경량으로 진행)"를 선택한 경우:

```
user_override = true
→ REVISION 이벤트에 반드시 기록
→ 사용자에게 경고 표시 (일관성 저하 가능)
→ 산출물 부록 A에 ※ 주의 표시
```

경고 문구:
```
⚠️ 전체 재컨설팅 없이 경량 진행합니다.
   결과 일관성이 낮을 수 있으며 최종 산출물에 기록됩니다.
```

## 판정 순서도

```
피드백 수신
    │
    ▼
설명/질문만인가?
    YES → 타입 C
    NO  ↓
    │
    ▼
솔루션 ID(핵심 MS 제품 조합)가 바뀌는가?
    YES → 타입 B (전체 재컨설팅 권장, 사용자 확인)
    NO  → 타입 A (경량 재컨설팅)
    │
    ▼
판단 불명확?
    → 질문으로 명확화 후 재판정
    → 명확화 불가 시 보수적으로 타입 B 적용
```
